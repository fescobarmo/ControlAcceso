### 1. Construcci√≥n de la Imagen Docker

- El **prefijo num√©rico** (`01-`, `02-`) garantiza la ejecuci√≥n secuencial alfab√©tica

- `02-init-db.js` despu√©s puede ejecutar l√≥gica adicional si es necesario
- PostgreSQL ejecuta estos scripts **solo si la base de datos est√° vac√≠a**

### 2. Primera Ejecuci√≥n del Contenedor

Cuando ejecutas `docker-compose up`, PostgreSQL detecta si existe el volumen de datos:

```yaml

  - postgres_data:/var/lib/postgresql/data
```

**Comportamiento:**
- Si el directorio `/var/lib/postgresql/data` est√° **vac√≠o** ‚Üí ejecuta scripts de inicializaci√≥n
- Si el directorio **contiene datos** ‚Üí omite la inicializaci√≥n (base de datos ya existe)

#### Verificaci√≥n del Healthcheck

```yaml
healthcheck:
  test: ["CMD-SHELL", "pg_isready -U ${DB_USER:-postgres} -d ${DB_NAME:-controlacceso}"]
  interval: 10s
  timeout: 5s
  retries: 5
  start_period: 30s
```

**Flujo del healthcheck:**
1. Espera 30 segundos inicial (`start_period`)
2. Ejecuta `pg_isready` cada 10 segundos
3. Requiere 5 checks exitosos consecutivos
4. Una vez "healthy" ‚Üí permite que el backend inicie

##### Trigger 2: Auditor√≠a Autom√°tica

```sql
RETURNS TRIGGER AS $$
BEGIN
    IF TG_OP = 'INSERT' THEN
        INSERT INTO auditoria (usuario_id, accion, tabla_afectada, registro_id, datos_nuevos)
        VALUES (current_setting('app.current_user_id', true)::integer, 'INSERT', TG_TABLE_NAME, NEW.id, to_jsonb(NEW));
        RETURN NEW;
    ELSIF TG_OP = 'UPDATE' THEN
        INSERT INTO auditoria (usuario_id, accion, tabla_afectada, registro_id, datos_anteriores, datos_nuevos)
        VALUES (current_setting('app.current_user_id', true)::integer, 'UPDATE', TG_TABLE_NAME, NEW.id, to_jsonb(OLD), to_jsonb(NEW));
        RETURN NEW;
    ELSIF TG_OP = 'DELETE' THEN
        INSERT INTO auditoria (usuario_id, accion, tabla_afectada, registro_id, datos_anteriores)
        VALUES (current_setting('app.current_user_id', true)::integer, 'DELETE', TG_TABLE_NAME, OLD.id, to_jsonb(OLD));
        RETURN OLD;
    END IF;
    RETURN NULL;
END;
$$ LANGUAGE plpgsql;
```

**Aplicado a:**
```sql

    FOR EACH ROW EXECUTE FUNCTION audit_trigger_function();
```

**Qu√© registra:**

- **UPDATE:** Datos antes y despu√©s del cambio

```sql
CREATE OR REPLACE FUNCTION check_user_access(
    p_user_id INTEGER,
    p_area_id INTEGER,
    p_tipo_acceso VARCHAR DEFAULT 'lectura'
)
RETURNS BOOLEAN AS $$
DECLARE
    v_permiso VARCHAR;
    v_hora_actual TIME;
    v_dia_semana INTEGER;
BEGIN
    -- Obtener permiso del usuario
    SELECT tipo_permiso INTO v_permiso
    FROM permisos_acceso
    WHERE usuario_id = p_user_id 
    AND area_id = p_area_id
    AND is_active = true
    AND (fecha_fin IS NULL OR fecha_fin >= CURRENT_DATE);
    
    IF v_permiso IS NULL THEN
        RETURN FALSE;
    END IF;
    
    -- Verificar horario
    v_hora_actual := CURRENT_TIME;
    v_dia_semana := EXTRACT(DOW FROM CURRENT_DATE) + 1;
    
    -- Verificar si est√° en el horario permitido
    SELECT EXISTS(
        SELECT 1 FROM permisos_acceso
        WHERE usuario_id = p_user_id 
        AND area_id = p_area_id
        AND is_active = true
        AND (fecha_fin IS NULL OR fecha_fin >= CURRENT_DATE)
        AND v_hora_actual BETWEEN horario_inicio AND horario_fin
        AND v_dia_semana = ANY(dias_semana)
    ) INTO v_permiso;
    
    RETURN v_permiso;
END;
$$ LANGUAGE plpgsql;
```

**Uso:**

```sql
-- Retorna: true o false
```

**L√≥gica implementada:**
1. ¬øTiene permiso activo? ‚Üí Si no, retorna `false`
2. ¬øEst√° dentro del horario permitido? ‚Üí Verifica hora actual
3. ¬øEs un d√≠a permitido? ‚Üí Verifica d√≠a de la semana

## ‚ö° Healthcheck y Verificaci√≥n

El healthcheck garantiza que la base de datos est√° lista antes de que otros servicios inicien:

```yaml
healthcheck:
  test: ["CMD-SHELL", "pg_isready -U ${DB_USER:-postgres} -d ${DB_NAME:-controlacceso}"]
  interval: 10s
  timeout: 5s
  retries: 5
  start_period: 30s
```

### Par√°metros explicados:

- **test:** Comando que verifica el estado
  - `pg_isready` es una utilidad de PostgreSQL que verifica si el servidor acepta conexiones
  
- **interval:** 10 segundos entre cada check
  
- **timeout:** 5 segundos m√°ximo para cada check
  
- **retries:** 5 intentos fallidos antes de marcar como "unhealthy"
  
- **start_period:** 30 segundos de "gracia" inicial
  - Durante este tiempo, los fallos no cuentan para los retries
  - Permite que PostgreSQL termine de inicializar

### Dependencias en docker-compose.yml:

```yaml
backend:
  depends_on:
    database:
      condition: service_healthy  # ‚Üê Espera a que database est√© "healthy"
```

**Flujo:**
1. Database inicia ‚Üí Estado: "starting"
2. Espera 30 segundos (start_period)
3. Ejecuta `pg_isready` cada 10 segundos
4. Despu√©s de 5 checks exitosos ‚Üí Estado: "healthy"
5. Backend puede iniciar

‚úÖ **Portabilidad:** Mueve el directorio = mueves los datos  
‚úÖ **Supervivencia:** Los datos persisten aunque borres los contenedores  

### Desventajas:

‚ùå **Permisos:** Problemas potenciales en sistemas Unix  
‚ùå **Rendimiento:** Ligeramente m√°s lento que vol√∫menes nativos Docker  
‚ùå **Dependencia de ruta:** Depende de la estructura del proyecto
